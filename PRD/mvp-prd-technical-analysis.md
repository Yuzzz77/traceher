# TraceHer MVP PRD — 技术 & 产品分析

> 分析日期：2026-05-26
> 基于文档：mvp-prd.md v1.0

---

## 1. 架构与逻辑亮点

### 1.1 二维码状态解耦（生成 ≠ 生效）

PRD 中最强的架构决策，是把二维码的**生成**与**激活**干净地分离。通过引入独立的 `qr_status` 状态机（`INACTIVE → ACTIVE → FROZEN → REVOKED`），与 `product_batch.batch_status` 互不耦合，同时达成了两个目标：

- **运营灵活性**：二维码可以在封装环节提前生成并贴码（制造侧关注），而激活由质检结果驱动（品控侧关注）。两方互不阻塞。
- **消费者体验的优雅降级**：扫到 `INACTIVE` 码返回的是结构化的、有明确语义的提示（"暂未完成质检，请稍后再试"），而不是 404 或空数据。在面向消费者的平台中，**数据的缺失**会被解读为"平台不可靠"，但**明确的状态**会被解读为"流程透明"。

PRD 正确地规避了最致命的信任崩塌场景：消费者扫码后发现空数据，一次交互即可永久损害平台可信度。

### 1.2 N:M 批次-原料关联 + `usage_ratio`

`batch_material_rel` 桥接表的选择完全符合实际制造业场景。真实的生产从来不是干净的 1:1 对应——一批卫生巾可能同时消耗 A 批棉花、B 批无纺布、C 批 SAP 高分子吸水材料；而同一批原料也可能供应多个产品批次。`usage_ratio` 字段更进一步支持**比例化影响分析**：如果某原料仅占产品 5% 的成分且被标记风险，系统可以给出有区分度的结论，而非简单的二元安全/不安全判定。

此表也是整个逆向追溯承诺的唯一支撑点（第 4.2 节）。没有它，"投诉 → 产品批次 → 原料批次 → 受影响的其他产品批次"这条链路不成立。

### 1.3 EAV 风格的检测结果存储（`inspection_result`）

将每个检测项建模为一行（`item_name`, `result_value`, `result_level`, `standard_range`），而不是在 `inspection_report` 上横向扩展列，这对 MVP 是正确的：

- **Schema 稳定性**：新增检测项（如未来法规要求的"二噁英残留"）零 DDL。
- **精准查询**："查找所有荧光剂 = FAIL 的批次"是 `item_name + result_level` 上的索引查询，而非横跨数十个可空列的 WHERE 条件。
- **前端渲染**：扫码响应中的 `safety_results` 数组与该表 1:1 映射，API 层几无转换成本。

此方案的代价（失去列级类型约束）在 MVP 规模下可接受，可通过应用层校验补偿。

### 1.4 "坏的也要展示"哲学（第 2.4 节）

不合格批次展示明确的风险提示（含 `risk_reason` 和客服热线），而非隐藏或显示通用错误，且冻结状态的扫描响应是 API 设计中的一等公民，不是事后补丁。很多系统因为只设计了正常路径，出了问题只能把内部状态（堆栈、通用报错）泄露给消费者端，TraceHer 从设计层面避免了这个坑。

### 1.5 四柱信任模型

`信任 = 系统规则校验 + 人工复核 + 数据交叉验证 + 可追溯链路` 这个公式为每个功能提供了清晰的评审标准。四根支柱分别对应：自动校验规则引擎、人工审核队列、检测机构白名单/抽查机制、N:M 关系逆向追溯链。这使得整个架构可以对照信任模型直接进行审计。

## 2. 潜在边界情况与业务风险

### 2.1 上游风险等级追溯升级（供应商风险事后升高）

**缺口**：PRD 定义了 `supplier.risk_level`（NORMAL / WARNING / HIGH）并提及"高风险供应商加强人工复核频率"。但**没有任何事件驱动的追溯级联机制**。考虑以下场景：

1. 供应商 S 初始 `risk_level = NORMAL`，其棉批次 CB-001 通过检测。使用 CB-001 的产品批次 PB-001 被标记为 `VERIFIED`，二维码 `ACTIVE`。
2. 三周后，举报或抽查发现供应商 S 长期伪造农药检测报告。审核员将 `supplier.risk_level` 更新为 `HIGH`。
3. **PB-001 仍然 ACTIVE。** 消费者扫到这些码依然看到"已认证"——该产品实际上已经"继承"了一个已知高风险供应商的原料，却无任何预警。

`product_batch` 和 `qr_code` 表没有任何字段引用 `supplier.risk_level`，也没有触发器或工作流将供应商风险变更级联到下游实体。

**建议**：新增 `SupplierRiskEscalationEvent`，至少将包含该供应商原料的所有 ACTIVE 产品批次转换到一个 `REVIEW_PENDING` 子状态，触发复检。MVP 规模下可放到同步事务中处理；规模化后用异步事件 + 补偿审计追踪。`operation_log` 表已存在，可以直接记录此次级联操作。

### 2.2 单一审核员风险（人工复核的单点失控）

**缺口**：`inspection_report` 表只有一个 `auditor_id` 字段。审核流程（第 4.3 节）允许一名审核员独立完成 `PENDING → APPROVED` 的转换。系统缺少：

- 高风险品类**双人审批**要求
- **审核-被审核方隔离**（审核员不能审核自己所在供应商提交的报告）
- 争议报告的**仲裁机制**

一个被收买的审核员可以单方面批准一份伪造的检测报告（如实际甲醛严重超标的批次），系统会将其标记为 `VERIFIED` 并激活二维码。`operation_log` 会记录此操作，但发现时已是事后取证，不是事前防控。

**建议**：MVP 阶段至少实现轻量的**利益冲突规则**：审核员不得审核与其同属一个供应商主体提交的报告。产品化路线中，在 `inspection_report` 上增加 `required_approval_count` 字段（低风险默认为 1，高风险或首次提交默认为 2），在状态机层面强制执行——必须凑齐票数才能转入 `APPROVED`。

### 2.3 部分原料污染策略（N 个原料中 1 个有问题）

**缺口**：PRD 明确了两个追溯路径：
- **正向**："同一问题原料影响多个产品批次 → 批量标记风险 → 通知消费者"（第 4.2 节）
- **逆向**："投诉 → 产品批次 → 原料批次 → 供应商"（第 4.2 节）

但**没有定义**：*产品批次 A 使用了原料 X（合格）和原料 Y（不合格）——产品批次 A 和它的二维码应该怎么办？*

| 策略 | 行为 | 风险 |
|------|------|------|
| **保守（立即冻结）** | 任何 FAIL 原料 → 整批 FROZEN | 误报率高，供应商纠纷，库存浪费 |
| **比例（标记复核）** | 批次标记 WARNING，排队等审核员决定 | 存在消费者扫到潜在不安全产品的时间窗口 |
| **阈值制** | FAIL 原料的 `usage_ratio` > X% 才冻结，否则警告 | 增加配置复杂度，需要行业数据校准阈值 |

对于接触黏膜的卫生用品，保守策略作为默认值可能是正确的——但这必须是一个**显式的、有文档记录的设计决策**，而不是系统在运行时"自然呈现的行为"。

**建议**：在系统配置或硬编码规则中写入：**"任何关联原料批次 `risk_status = FROZEN`，则产品批次立即转入 FROZEN。`risk_reason` 字段关联所有不合格原料的全部 FAIL 检测项。"** 这条规则无歧义、安全优先。

### 2.4 二维码防伪能力不足

**缺口**：PRD 仅说明 `qr_code_value` 为"加密生成"，且实体关系中标为 `product_batch 1:1 qr_code`。这暗示**一个批次的所有产品包装共用一个二维码**。如果属实，这是防伪的根本性弱点：

- 造假者拿到一包正品，复制其二维码，印在 10 万包假货上。
- 所有扫码都指向同一个合法批次——与正品无法区分。
- 不存在的单品级追踪，无法检测异常扫码模式。

实体关系图写的是 `product_batch 1:1 qr_code`，但实际生产需要 `product_batch 1:N qr_code`——每个独立包装有一个唯一的、密码学随机的标识符，在批次命名空间内唯一。

**建议**：将关系改为 1:N。每个二维码使用 UUID 级唯一值。在 `qr_code` 表增加 `scan_count` 和 `first_scanned_at` 字段用于基础异常检测（同一码从 20 个城市被扫了 500 次 = 疑似仿冒标记）。此变更应在 Phase 1 schema 设计阶段就落地，即使异常检测逻辑可以推迟——因为事后改 1:1 到 1:N 是破坏性迁移，成本极高。

### 2.5 状态转换的竞态条件

**缺口**：所有带状态的表（`qr_code`, `product_batch`, `inspection_report`, `raw_material_batch`）均无 version 字段或乐观锁机制。以下并发操作可能发生：

- 审核员 A 点击"通过"报告 R，同时管理员冻结与 R 关联的批次 B。
- 系统自动校验将报告标记为 APPROVED，同时审核员正在驳回同一份报告。

没有 `version` 或 `updated_at` 级 compare-and-swap，这些冲突将以不确定的方式解决（取决于事务提交顺序），且没有任何冲突本身的审计记录。

**建议**：所有带状态转换的表均增加 `version`（INT，每次更新 +1）字段。所有状态转换使用 `UPDATE ... WHERE id = ? AND version = ?`，不匹配时显式失败或重试。每个表一行 DDL，预防一整类 Bug。

### 2.6 INACTIVE 码信息泄露风险

**缺口**：INACTIVE 的扫码响应很精简（`"message": "该产品暂未完成质检，请稍后再试"`），但 PRD 未明确对于未激活的码，是否应展示产品名称或批次号。如果展示了产品信息，竞争对手可以挨个扫货架上的产品，拼出一张品牌未发布产品和生产节奏的完整地图。如果不展示，消费者误扫到未上市产品时看到的是模糊提示，可能怀疑是假货。

这是一个涉及安全的产品决策，需要在 PRD 中明确。

## 3. 数据模型与 API 评审

### 3.1 Schema 评估

**`batch_material_rel` — 缺少关键索引**

逆向追溯查询（"哪些产品批次使用了原料批次 X？"）的 SQL 为：

```sql
SELECT pb.* FROM product_batch pb
JOIN batch_material_rel bmr ON pb.id = bmr.product_batch_id
WHERE bmr.raw_material_batch_id = ?;
```

如果没有 `(raw_material_batch_id, product_batch_id)` 上的联合索引，这将退化为关联表的全表扫描。这是召回工作流中延迟最敏感的查询——当某个原料被标记为问题原料时，系统必须立即找出所有受影响的产品批次。

**`inspection_result` — 联结深度**

消费者扫码接口需要：
```
qr_code → product_batch → inspection_report（最新 APPROVED） → inspection_result（全部行）
```

这是一个四表联结。MVP 规模（数千批次，非数百万）下配合适当索引没问题。但需要注意 `inspection_report` 会随时间累积多条记录（被驳回后重新提交）。查询必须取**最新 APPROVED** 的那份报告，意味着需要子查询 `WHERE report_status = 'APPROVED' ORDER BY audit_time DESC LIMIT 1`，或在 `product_batch` 上反范式的 `latest_approved_report_id` 字段。

**`product_batch` — 缺少 `supplier_id`**

当前到供应商的追溯路径是 5 跳：`qr_code → product_batch → batch_material_rel → raw_material_batch → supplier`。在该行业中，一个产品批次通常有一个主要原料供应商（即使 N:M 关联多种原料，棉花的供应商是主导），反范式的 `primary_supplier_id` 可以将常规追溯缩减为 2 跳。这对 MVP 来说过早，但应列入性能路线图。

**`qr_code` 与 `product_batch` 的关系 — 1:1 应为 1:N**

如第 2.4 节所述，此关系必须为 1:N 才能实现有意义的防伪。如果 PRD 的"1:1"本意是"每个二维码只指向一个产品批次"（这本身是正确的——一个码不应该指向多个批次），那么标注有歧义。实际基数应为：

```
product_batch 1 ──── N qr_code
qr_code       N ──── 1 product_batch
```

这是标准且无争议的设计。当前 ER 图中的 1:1 标注应修正。

### 3.2 API 评审 — `/api/v1/verify/scan` 响应

**设计优秀的点：**

- 响应扁平、边界清晰——无嵌套分页、无冗余字段。小程序渲染只需一次网络请求，拿到数据直接展示。
- `safety_results` 是结构一致的对象数组，在 WXML 模板中遍历渲染极其简单。
- `certification` 包裹对象配合 `report_verified: true` 对关注第三方验证的消费者是有效的信任信号。
- FROZEN 响应包含 `risk_reason` 字符串和 `hotline` 热线——是用户可操作的实际信息，而非仅错误码。

**缺失字段：**

1. **`scan_event_id`**：本次扫码没有标识符。如果消费者立刻发起投诉，前端无法引用"刚才那次扫码"——只能引用二维码值本身。这使投诉与具体扫码事件的关联模糊。返回一个轻量 UUID 作为 `scan_id` 并存入 `verification_result` 表（当前 ER 图中存在但未定义表结构）可闭环此问题。

2. **`material_summary`（P2）**：PRD 将"棉花产地"归为 P2 辅助信息，但扫码响应中完全没有对应字段。最小化添加：
   ```json
   "material_summary": {
     "cotton_origin": "新疆",
     "supplier_verified": true
   }
   ```
   这两个字段可收入 UI 中"更多信息"折叠区域，不干扰核心信息展示。

3. **`shelf_life` / `expiration_date`**：消费者买卫生用品常规会看保质期。当前有 `production_date` 但没有保质期，消费者无法判定是否过期。增加 `expiration_date`（或按品类推导的 `shelf_life_months`）是一个字段变更，消费者效用很高。

4. **INACTIVE 响应的颗粒度**：当前 INACTIVE 响应只有一句通用提示。区分以下场景可减少消费者困惑：
   - `INACTIVE + batch_status = UNVERIFIED` → "该产品已生产，正在等待质检"
   - `INACTIVE + batch_status = null` → "该产品尚未进入质检流程"

5. **Cache-Control 头**：正常状态的扫码响应在 CDN 层面可缓存（产品信息不会每次扫码都变）。API 规范应包含 `Cache-Control: public, max-age=3600`（正常响应）和 `no-cache`（FROZEN/WARNING 响应）——这是 HTTP 层的关注点，PRD 未涉及。

## 4. 实施瓶颈与演进路线图

### 4.1 第一期瓶颈（核心验证链路）

**瓶颈 1：同步状态传播**

第一期技术风险最高的操作是冻结级联。当审核员冻结一个原料批次时，系统必须：

1. 查询 `batch_material_rel` 获得所有 `product_batch_id`
2. 逐一评估冻结策略（部分 vs. 全部污染——当前未定义）
3. 更新 `product_batch.batch_status = FROZEN`
4. 对每个冻结的产品批次，更新其下所有 `qr_code.qr_status = FROZEN`
5. 如果扫码响应有缓存，逐个失效
6. 所有流转写入 `operation_log`

如果实现为单个同步事务，锁将跨 4 张表持有。如果朴素实现（逐行循环更新），锁时长随受影响批次数量线性增长。**缓解方案**：在单个事务内使用批量 `UPDATE ... WHERE id IN (...)`，并设置 `statement_timeout` 防止失控锁。

**瓶颈 2：二维码批量生成**

PRD 将二维码生成视为已解决的问题，但密码学唯一性约束和与产品批次的 1:N 关系意味着生成管线必须：
- 为每个批次生成 N 个唯一、不可序、不可猜测的码（N 可能达到 10⁵ ~ 10⁶）
- 高效存储（100 字节 × 100 万 = 每个批次 100 MB 仅码值）
- 导出为打印系统可消费的格式

这是吞吐量问题而非正确性问题，但如果不仔细 scoping，会占据第一期大量工程时间。**缓解方案**：使用 `COPY`（PostgreSQL）或批量 `INSERT` 配合 UUID 生成方案入库；打印导出放到异步 Job 中处理。

### 4.2 第二期瓶颈（审核与风控）

**瓶颈 1：校验规则引擎的复杂度增长**

系统自动校验当前有 4 条规则（第 4.3 节）。自然的演进是每新增一个风险场景就新增一条规则。如果不在初期采用可插拔的规则架构，校验代码将退化为巨大的 if-else 链条。**缓解方案**：采用责任链模式——每条规则是独立的类/函数，统一接口为 `validate(report) -> Pass | Fail(reason)`。PRD 中 `inspection_result` 的 EAV 设计天然适配此模式。

**瓶颈 2：审核队列的任务分配**

PRD 指定了"人工复核队列"但未定义分配语义。缺少：
- **认领/锁定**：两位审核员同时打开同一份报告
- **负载均衡**：一位审核员拿到 50 份，另一位拿到 0 份
- **超时升级**：一份报告在队列里躺了 72 小时无人处理

**缓解方案**：在 `inspection_report` 增加 `claimed_by` 和 `claimed_at` 字段。实现"认领 → 复核 → 决定"工作流，带可配置的自动释放超时（如 `claimed_at` 超过 4 小时且 `report_status` 仍为 `PENDING`，则释放回队列）。

### 4.3 演进路线图：MVP → 生产级

#### 建议一：扫码接口的读写分离 / CQRS

扫码接口（`/api/v1/verify/scan`）将承载 95%+ 的流量。生产规模（百万消费者、每秒数千次扫码）下，每次请求都做四表联结不可持续。

**目标架构：**
```
写路径：  状态变更 → DB 事务 → 发布事件 → 构建读模型
读路径：  扫码请求 → Redis/Mongo 读模型（以 qr_code_value 为 key）→ 直接返回
```

读模型是预联结、反范式的 JSON 文档，与扫码响应结构完全一致。任何上游实体状态变更时重建（或局部更新）。这彻底解耦了写复杂度和读性能。MVP 设计中的实时 DB 查询在当前阶段是正确的——但读模型应列入 v1.1 路线图，且 API 契约必须保持稳定，确保后端替换对小程序不可见。

**落地步骤**：在第二期 schema 中增加 `scan_response_cache_key` 列或在 Redis 中定义规范的 key 命名约定，使 cache-aside 层有一个标准的失效 key。

#### 建议二：事件驱动的状态传播

当前设计隐含同步级联更新（冻结原料 → 冻结批次 → 冻结二维码）。规模化后这会产生事务扇出问题。

**目标架构：**
```
审核员冻结 RawMaterialBatch#42
  → RawMaterialBatch 聚合发布 RawMaterialFrozenEvent { materialBatchId: 42, reason: "..." }
  → ProductBatchProjection 订阅 → 查询 batch_material_rel → 发布 ProductBatchFrozenEvent(s)
  → QrCodeProjection 订阅 → 更新所有关联二维码
  → NotificationService 订阅 → 推送消费者预警
```

每个服务拥有自己的事务边界。系统是最终一致性的，意味着存在亚秒级的时间窗口——消费者可能扫到"本应已冻结但尚未完成更新"的码。**补偿措施**：扫码服务同时检查二维码自身状态**和**产品批次状态（轻量第二查询或反范式字段），作为安全兜底。

这是从模块化单体到领域事件驱动系统的架构升级，不适合 MVP 阶段。但 `operation_log` 表已经捕获了事件日志的原始素材——存在自然的迁移路径。

#### 建议三：单品级二维码 + 扫码异常检测

如第 2.4 节所述，`product_batch` 与 `qr_code` 的关系必须从 1:1 升为 1:N。除 schema 变更外，生产级架构还应包括：

- **二维码命名空间设计**：使用确定性前缀（如前 8 位编码批次 ID）+ 密码学随机后缀。这样无需 DB 查表即可识别批次，同时防止码值猜测。
- **扫码遥测管线**：每次扫码记录 `(qr_code_id, timestamp, ip_geolocation, user_agent)` 进入轻量事件流（或 `verification_result` 表）。每日批处理计算：
  - N 次以上扫码 + M 个以上不同位置 → 疑似仿冒标记
  - 激活后 90 天零扫码 → 供应链泄漏标记
- **扫码接口限流**：按 IP 和按二维码值的频率控制，防止爬取和暴力枚举。

## 总结

PRD 整体结构扎实，产品思维清晰。N:M 数据模型、二维码状态解耦、"坏的也要展示"这三点对于该领域都是正确的设计选择。以上识别的缺口集中在**运维安全层**——事后变更如何追溯、谁能批准什么、并发下系统行为如何收敛。在第一期实施前处理好这些点，可以避免一旦消费者开始扫码生产环境就很难返工修复的架构债务。
